# النشر والمراقبة — دليل مختصر

## 1) قرار المنتج: خادم أم لا

| الوضع | متى يناسبكم | ماذا يعني للمستخدم |
|--------|-------------|---------------------|
| **بدون خادم** (GitHub Pages + ملفات ثابتة) | عرض، تجربة، demo | الحسابات واللوحة في **متصفح الجهاز فقط**؛ المزامنة بين أجهزة عبر **تصدير/استيراد JSON** من لوحة التحكم. |
| **مع خادم** | إنتاج، فريق أدمن، تتبع مستخدمين مركزي | API + قاعدة بيانات (أو خدمة مثل Supabase)؛ يحتاج مفاتيح، تكلفة، وخصوصية/امتثال. |

الكود الحالي مُحسَّن للوضع **الثابت** مع تنبيهات واضحة في الواجهة.

---

## 2) GitHub Pages

1. **Settings → Pages → Source:** GitHub Actions (وليس Branch قديم).
2. بعد كل `push` إلى `main`: راقب workflow **Deploy GitHub Pages** (نجاح/فشل).
3. الرابط النموذجي: `https://<user>.github.io/<repo>/` — راجع أيضاً [VERCEL-DEPLOY.md](./VERCEL-DEPLOY.md) وروابطكم الفعلية.

---

## 3) Vercel (إن استخدمتم نشر الواجهة من نفس المجلد)

1. **Root Directory:** `financial-consulting/iif-fund-demo`.
2. **Build Command:** الأفضل تركه من **`vercel.json`** (`npm run i18n:packs`). إن عيّنتم أمراً يدوياً في اللوحة، يجب أن يشمل **توليد حزم i18n**.
3. بعد النشر: افتحوا الموقع وتحققوا من قسم حرج (مثلاً اللغة أو لوحة التحكم).

---

## 4) المراقبة اليومية (بسيطة)

- **Actions → CI:** يجب أن يمر **verify** (بناء + فحوص + `verify:i18n-packs-sync` عند التفعيل).
- إن فشل **`verify:i18n-packs-sync`:** شغّلوا من جذر الريبو `npm run i18n:packs` ثم ارفعوا ملفات `i18n-service-packs-*.js`.
- **Uptime / health:** لديكم workflows مثل `uptime-ping` إن كانت مفعّلة — راجعوا النتائج دورياً.

---

## 5) فحص تلقائي محلي / CI

```bash
npm run check:deploy
```

يتحقق من وجود ملفات النشر الحرجة ومنطقية `vercel.json` (بدون الاتصال بـ Vercel).

---

## 6) إن قررتم لاحقاً إضافة خادم

1. اختاروا منصة (مثلاً Supabase، أو API على Vercel + Postgres).
2. **لا تضعوا أسرار الإنتاج في Git** — استخدموا متغيرات البيئة في المنصة فقط.
3. صمموا مسارات واضحة: تسجيل، جلسة، قائمة مستخدمين للأدمن — ثم اربطوا الواجهة تدريجياً بدل الاعتماد الكامل على `localStorage`.

نموذج متغيرات اختيارية (مستقبلية) — انظر [.env.example](./.env.example).

### 6.1) ويبهوك تسجيل (Vercel — اختياري)

بعد **إنشاء حساب جديد** من الواجهة، المتصفح يرسل طلباً إلى `POST /api/user-registration-forward` على نفس النطاق (لا يؤثر على GitHub Pages إن لم يكن هناك API).

- **بدون إعداد:** الاستجابة `skipped` — لا خطأ في الواجهة.
- **مع إعداد:** في لوحة Vercel → Environment Variables:
  - `IIF_USER_WEBHOOK_URL` — عنوان **HTTPS** فقط (مثلاً Zapier/Make أو API داخلي).
  - `IIF_USER_WEBHOOK_SECRET` (اختياري) — يُرسل كـ `Authorization: Bearer …` للويبهوك.

الحمولة المرسلة للويبهوك: `{ email, name, at, source }` (بدون كلمة مرور).
